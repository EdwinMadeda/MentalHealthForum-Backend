2. Activity Tracking (last_active_at) ⚠️ NOT STARTED
   Description
   Populate the app_users.last_active_at timestamp to enable "last active" sorting and display in connections, member discovery, and global search results.

Why This Matters
last_active_at is already in the AppUserRepository sorting options (sortBy = 'last_active_at').

Without updates, the sort order is NULL-heavy → poor UX.

Activity tracking is essential for "active now" indicators in user connections.

Architecture Decision: Rate-Limited Updates (Anti-Write-Storm Pattern)
Do NOT update on every request. Use a cooldown period (e.g., 5 minutes) to avoid write storms.

Implementation Steps
1. Add a tracking column:

sql
ALTER TABLE app_users ADD COLUMN last_active_updated_at TIMESTAMP DEFAULT NULL;
2. Create a partial index for performance:

sql
CREATE INDEX idx_active_users ON app_users (last_active_at) WHERE is_active = TRUE;
3. In a WebFilter/Interceptor (Reactive), implement the rate-limited update:

java
// Java/Spring WebFlux example
@Component
public class ActivityTrackingFilter implements WebFilter {
private static final Duration UPDATE_THRESHOLD = Duration.ofMinutes(5);
private final UserRepository userRepository;
private final Cache<String, Instant> activityCache; // Optional, for even fewer DB hits

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        return chain.filter(exchange)
            .then(Mono.defer(() -> updateActivityIfNeeded(exchange)))
            .then(); // or .then(chain.filter(exchange)) depending on placement
    }

    private Mono<Void> updateActivityIfNeeded(ServerWebExchange exchange) {
        String userId = getAuthenticatedUserId(exchange);
        if (userId == null) return Mono.empty();

        return userRepository.findById(UUID.fromString(userId))
            .filter(user -> shouldUpdate(user.getLastActiveUpdatedAt()))
            .flatMap(user -> userRepository.updateLastActive(userId, Instant.now()))
            .then();
    }

    private boolean shouldUpdate(Instant lastUpdated) {
        if (lastUpdated == null) return true;
        return Duration.between(lastUpdated, Instant.now()).compareTo(UPDATE_THRESHOLD) >= 0;
    }
}

3. Demotion + Deactivation Transaction Control ⚠️ NOT STARTED
   Description
   Prevent admins/moderators from being deactivated in the same transaction where they are demoted. This ensures the trigger's NEW check doesn't bypass the deactivation protection.

Implementation Approach
In the Backend Service Layer:

java
// UserModerationService or AdminService
@Transactional
public Mono<Void> demoteAndDeactivate(UUID userId) {
// Step 1: Demote (remove admin/mod roles/groups)
return appUserRepository.demoteUser(userId)
// Step 2: Deactivate (only after demotion is committed)
.then(appUserRepository.deactivateUser(userId))
.as(transactionalOperator::transactional);

Important:

Ensure Step 1 (demote) is committed before Step 2 (deactivate) executes.

Use Mono.defer() or Mono.fromCallable() to sequence them explicitly.

If Step 2 fails, the demotion should roll back (or be logged) – use the @Transactional propagation correctly.
